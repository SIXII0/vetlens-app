## 问题一（续）：Tesseract.js 浏览器端 OCR 返回"未知错误"

### 现象
服务端 PaddleOCR 不可用 → 降级到 Tesseract.js → 报错 "Tesseract OCR 失败: 未知错误"

### 根因分析 (第二轮排查)

原代码三个问题叠加：

1. **错误捕获信息丢失**：`catch` 块只处理 `Error` 类型，但 Tesseract.js 抛出的异常可能是字符串或普通对象
   ```typescript
   // 旧代码
   catch (err) {
     error: `失败: ${err instanceof Error ? err.message : '未知错误'}`
   }
   // → err 是对象时只显示 "未知错误"，用户看不到真实原因
   ```

2. **图片传入方式不可靠**：`URL.createObjectURL(file)` 产生 blob URL，Tesseract.js v5 对其支持不稳定

3. **语言包下载可能超时**：Tesseract.js 默认从 jsdelivr CDN 下载 `chi_sim` 语言包（~15MB），国内网络可能超时

### 修复内容

| 修复 | 文件 | 说明 |
|------|------|------|
| 错误日志增强 | `ocr-client.ts` | `catch (err: unknown)` + `String(err)` + `JSON.stringify(err)` — 无论什么类型都能显示 |
| 网络错误识别 | `ocr-client.ts` | 检测 `NetworkError` / `timeout` / `Failed to fetch`，给出针对性的网络提示 |
| WASM 检测 | `ocr-client.ts` | 检测 WebAssembly 支持问题，提示用 Chrome/Edge |
| 图片传入改为 canvas | `ocr-client.ts` | `File → FileReader → HTMLImageElement` 转成标准 Image 对象再传 Tesseract |
| Worker 安全释放 | `ocr-client.ts` | `finally` 块确保 `worker.terminate()` |
| UI 错误详情 | `upload/+page.svelte` | 显示 `errorDetail` 字段，用户能看到完整错误信息 |

### 验证方法
1. 打开浏览器 DevTools → Console
2. 上传图片，观察 `[Tesseract]` 开头的日志：
   - `开始初始化 worker...` → `加载核心...` → `下载语言包...` → `识别中...`
3. 如果卡在"下载语言包"，说明 CDN 不通——需要 VPN 或等 Tesseract 下载完成（首次 ~15MB）
4. 如果日志显示具体错误信息，可根据错误类型排查

## 问题一：上传医疗报告无法自动识别填充

### 现象
用户在 `/upload` 页面上传账单/医疗报告照片后，OCR 无法自动识别文字内容，只能手动输入。

### 根因分析

```
用户上传图片
  → processOcr() 调用 /api/ocr
    → 服务端 performOcr()
      → preprocessImage() ✅ 正常
      → isPaddleAvailable() → ❌ PaddleOCR Docker 未运行
      → 返回 { success: false, engine: 'manual' }
    → 前端收到失败结果 → 切换到空表单（手动输入）
```

**三个层面的问题**：

| 层级 | 问题 | 影响 |
|------|------|------|
| **架构** | 只实现了 PaddleOCR（需要 Docker），Tesseract.js 降级方案在设计文档中提及但从未编码 | 没有 Docker 就无法 OCR |
| **体验** | OCR 失败后直接给空表单，没有任何自动识别尝试 | 用户感知为"完全不能用" |
| **日志** | 静默失败，不告知用户失败原因 | 用户不知道是缺 Docker 还是图片问题 |

### 修复内容

1. **新增** `src/lib/utils/ocr-client.ts` — 浏览器端 Tesseract.js OCR 客户端
   - 使用 `tesseract.js@5` (WASM)，无需 Docker
   - 支持中文 + 英文混合识别 (`chi_sim+eng`)
   - 包含与服务端一致的结构化文本提取逻辑
   - 检测浏览器 WASM 可用性 `isTesseractAvailable()`

2. **修改** `src/routes/upload/+page.svelte` — 三级 OCR 降级策略
   ```
   策略一: PaddleOCR (服务端, Docker)
     ↓ 不可用
   策略二: Tesseract.js (浏览器端, WASM)
     ↓ 不可用
   策略三: 手动输入表单
   ```
   - 每级失败时显示明确的引擎名称和失败原因
   - Tesseract.js 识别成功时显示置信度

3. **安装** `tesseract.js@5` 依赖

### 验证方法
1. 启动应用 `npm run dev`
2. 访问 `/upload`，上传一张中文账单图片
3. 观察 OCR 进度: 会先尝试服务端 → 自动降级到浏览器端 Tesseract.js → 识别成功
4. 如 Tesseract 也失败，检查浏览器 WASM 支持和图片清晰度

## 问题二：账单没有分析结果

### 现象
用户手动输入账单项目后点击"开始解读"，跳转到分析结果页但内容为空或无匹配结果。

### 根因分析

```
用户点击"开始解读"
  → POST /api/analyze
    → analyzeBill()
      → matchTerm("血常规")
        → matchFTS5() → searchTerms() → FTS5 MATCH 查询
```

**三个层面的问题**：

| 层级 | 问题 | 影响 |
|------|------|------|
| **FTS5 索引** | 术语导入时依赖 TRIGGER 自动同步 FTS5，但 `INSERT OR REPLACE` 可能导致 TRIGGER 不触发（REPLACE = DELETE + INSERT，但 DELETE 在 REPLACE 前的 DELETE 不会触发 AFTER INSERT） | FTS5 索引可能为空 → 所有查询返回空 |
| **FTS5 查询** | 对中文短词构建的 FTS5 查询不够鲁棒，`split(/\s+/)` 对中文不拆分；无 FTS5 失败后的回退机制 | 中文词查询返回空 → `searchTerms` catch 后返回 `[]` |
| **匹配链断裂** | FTS5 返回空→matchFTS5 返回空→matchLevenshtein 阈值 0.4 过高→全部失败→所有项目 isUnknown=true | 所有项目显示"未知"，解释只有通用模板 |

### 修复内容

1. **重写** `src/lib/server/db/terms.ts` — FTS5 搜索增强
   - **索引同步修复**: `insertTerms` 完成后主动调用 `rebuildFtsIndex()` 直接写入 FTS5（不再依赖 TRIGGER）
   - **查询构建优化**: `buildFts5Query()` 区分中英文 — 中文短语用 `"血常规"` 加前缀匹配 `"血"*`，英文分词用 OR
   - **前缀匹配**: 短词自动添加 `*` 通配符
   - **LIKE 回退**: 新增 `fallbackSearch()` — FTS5 为空或查询失败时自动使用 `LIKE '%keyword%'` 搜索
   - **FTS 计数检查**: 先检查 FTS 表是否有数据，为空时跳过 FTS5 使用 LIKE

2. **重写** `src/lib/server/engine/matcher.ts` — 匹配引擎增强
   - **子串匹配加分**: 增加 `containsSubstring` 检查，匹配到子串时置信度 +0.2
   - **阈值优化**: FTS5 从 0.7 降至 0.6；Levenshtein 从 0.4 降至 0.35
   - **兜底机制**: 即使低置信度也返回最佳匹配（而非 null）
   - **详细日志**: 每级匹配都输出日志，便于排查

3. **增强** `src/routes/api/analyze/+server.ts`
   - 添加请求计时
   - 添加匹配统计日志
   - 数据库写入失败不阻止分析结果返回
   - 过滤空项目

4. **增强** `src/routes/analysis/[id]/+page.svelte`
   - 添加分析摘要面板（匹配/未匹配/价格偏高统计）
   - 添加空项目状态提示
   - 改进分类颜色映射

### 验证方法
1. 重新构建后访问应用
2. 在控制台查看 `[VetLens]` 日志 — 应看到 FTS5 加载成功的消息
3. 上传账单或手动输入"血常规 ¥120" → 点击解读
4. 结果页应显示 ✅ 已匹配（置信度 > 60%），解释内容非空

## 问题三：Docker / OCR 引擎环境搭建

### 环境检测结果

| 组件 | 状态 | 详情 |
|------|------|------|
| Docker Desktop | ❌ 未安装 | WSL2 可用 (v2.4.13) 但 Docker 未安装 |
| PaddleOCR | ❌ 不可用 | PaddlePaddle 3.3.1 + Python 3.13 存在 PyTorch DLL 兼容问题 |
| EasyOCR | ❌ 不可用 | 模型托管在 GitHub，国内下载超时 |
| Tesseract OCR | ❌ 未安装 | winget 安装也走 GitHub，同样下载慢 |

### 当前方案

应用**核心功能完全可用**（手动输入 + 分析引擎 + 所有管理功能）。

**启用 OCR 自动识别**需安装 Tesseract OCR：

1. 下载 [tesseract-ocr-w64-setup-5.4.0.exe](https://github.com/UB-Mannheim/tesseract/releases)
2. 安装时勾选 **Chinese Simplified** 语言包
3. 双击 `ocr_server/start-ocr.bat` 启动 OCR 服务
4. 上传图片即可自动识别

安装一次永久可用，无需 Docker。
