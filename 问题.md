## 问题一（续）：Tesseract.js 浏览器端 OCR 返回"未知错误"

### 现象
服务端 PaddleOCR 不可用 → 降级到 Tesseract.js → 报错 "Tesseract OCR 失败: 未知错误"

### 根因分析 (第二轮排查)

原代码三个问题叠加：

1. **错误捕获信息丢失**：`catch` 块只处理 `Error` 类型，但 Tesseract.js 抛出的异常可能是字符串或普通对象
   ```typescript
   // 旧代码
   catch (err) {
     error: `失败: ${err instanceof Error ? err.message : '未知错误'}`
   }
   // → err 是对象时只显示 "未知错误"，用户看不到真实原因
   ```

2. **图片传入方式不可靠**：`URL.createObjectURL(file)` 产生 blob URL，Tesseract.js v5 对其支持不稳定

3. **语言包下载可能超时**：Tesseract.js 默认从 jsdelivr CDN 下载 `chi_sim` 语言包（~15MB），国内网络可能超时

### 修复内容

| 修复 | 文件 | 说明 |
|------|------|------|
| 错误日志增强 | `ocr-client.ts` | `catch (err: unknown)` + `String(err)` + `JSON.stringify(err)` — 无论什么类型都能显示 |
| 网络错误识别 | `ocr-client.ts` | 检测 `NetworkError` / `timeout` / `Failed to fetch`，给出针对性的网络提示 |
| WASM 检测 | `ocr-client.ts` | 检测 WebAssembly 支持问题，提示用 Chrome/Edge |
| 图片传入改为 canvas | `ocr-client.ts` | `File → FileReader → HTMLImageElement` 转成标准 Image 对象再传 Tesseract |
| Worker 安全释放 | `ocr-client.ts` | `finally` 块确保 `worker.terminate()` |
| UI 错误详情 | `upload/+page.svelte` | 显示 `errorDetail` 字段，用户能看到完整错误信息 |

### 验证方法
1. 打开浏览器 DevTools → Console
2. 上传图片，观察 `[Tesseract]` 开头的日志：
   - `开始初始化 worker...` → `加载核心...` → `下载语言包...` → `识别中...`
3. 如果卡在"下载语言包"，说明 CDN 不通——需要 VPN 或等 Tesseract 下载完成（首次 ~15MB）
4. 如果日志显示具体错误信息，可根据错误类型排查

## 问题一：上传医疗报告无法自动识别填充

### 现象
用户在 `/upload` 页面上传账单/医疗报告照片后，OCR 无法自动识别文字内容，只能手动输入。

### 根因分析

```
用户上传图片
  → processOcr() 调用 /api/ocr
    → 服务端 performOcr()
      → preprocessImage() ✅ 正常
      → isPaddleAvailable() → ❌ PaddleOCR Docker 未运行
      → 返回 { success: false, engine: 'manual' }
    → 前端收到失败结果 → 切换到空表单（手动输入）
```

**三个层面的问题**：

| 层级 | 问题 | 影响 |
|------|------|------|
| **架构** | 只实现了 PaddleOCR（需要 Docker），Tesseract.js 降级方案在设计文档中提及但从未编码 | 没有 Docker 就无法 OCR |
| **体验** | OCR 失败后直接给空表单，没有任何自动识别尝试 | 用户感知为"完全不能用" |
| **日志** | 静默失败，不告知用户失败原因 | 用户不知道是缺 Docker 还是图片问题 |

### 修复内容

1. **新增** `src/lib/utils/ocr-client.ts` — 浏览器端 Tesseract.js OCR 客户端
   - 使用 `tesseract.js@5` (WASM)，无需 Docker
   - 支持中文 + 英文混合识别 (`chi_sim+eng`)
   - 包含与服务端一致的结构化文本提取逻辑
   - 检测浏览器 WASM 可用性 `isTesseractAvailable()`

2. **修改** `src/routes/upload/+page.svelte` — 三级 OCR 降级策略
   ```
   策略一: PaddleOCR (服务端, Docker)
     ↓ 不可用
   策略二: Tesseract.js (浏览器端, WASM)
     ↓ 不可用
   策略三: 手动输入表单
   ```
   - 每级失败时显示明确的引擎名称和失败原因
   - Tesseract.js 识别成功时显示置信度

3. **安装** `tesseract.js@5` 依赖

### 验证方法
1. 启动应用 `npm run dev`
2. 访问 `/upload`，上传一张中文账单图片
3. 观察 OCR 进度: 会先尝试服务端 → 自动降级到浏览器端 Tesseract.js → 识别成功
4. 如 Tesseract 也失败，检查浏览器 WASM 支持和图片清晰度

## 问题二：账单没有分析结果

### 现象
用户手动输入账单项目后点击"开始解读"，跳转到分析结果页但内容为空或无匹配结果。

### 根因分析

```
用户点击"开始解读"
  → POST /api/analyze
    → analyzeBill()
      → matchTerm("血常规")
        → matchFTS5() → searchTerms() → FTS5 MATCH 查询
```

**三个层面的问题**：

| 层级 | 问题 | 影响 |
|------|------|------|
| **FTS5 索引** | 术语导入时依赖 TRIGGER 自动同步 FTS5，但 `INSERT OR REPLACE` 可能导致 TRIGGER 不触发（REPLACE = DELETE + INSERT，但 DELETE 在 REPLACE 前的 DELETE 不会触发 AFTER INSERT） | FTS5 索引可能为空 → 所有查询返回空 |
| **FTS5 查询** | 对中文短词构建的 FTS5 查询不够鲁棒，`split(/\s+/)` 对中文不拆分；无 FTS5 失败后的回退机制 | 中文词查询返回空 → `searchTerms` catch 后返回 `[]` |
| **匹配链断裂** | FTS5 返回空→matchFTS5 返回空→matchLevenshtein 阈值 0.4 过高→全部失败→所有项目 isUnknown=true | 所有项目显示"未知"，解释只有通用模板 |

### 修复内容

1. **重写** `src/lib/server/db/terms.ts` — FTS5 搜索增强
   - **索引同步修复**: `insertTerms` 完成后主动调用 `rebuildFtsIndex()` 直接写入 FTS5（不再依赖 TRIGGER）
   - **查询构建优化**: `buildFts5Query()` 区分中英文 — 中文短语用 `"血常规"` 加前缀匹配 `"血"*`，英文分词用 OR
   - **前缀匹配**: 短词自动添加 `*` 通配符
   - **LIKE 回退**: 新增 `fallbackSearch()` — FTS5 为空或查询失败时自动使用 `LIKE '%keyword%'` 搜索
   - **FTS 计数检查**: 先检查 FTS 表是否有数据，为空时跳过 FTS5 使用 LIKE

2. **重写** `src/lib/server/engine/matcher.ts` — 匹配引擎增强
   - **子串匹配加分**: 增加 `containsSubstring` 检查，匹配到子串时置信度 +0.2
   - **阈值优化**: FTS5 从 0.7 降至 0.6；Levenshtein 从 0.4 降至 0.35
   - **兜底机制**: 即使低置信度也返回最佳匹配（而非 null）
   - **详细日志**: 每级匹配都输出日志，便于排查

3. **增强** `src/routes/api/analyze/+server.ts`
   - 添加请求计时
   - 添加匹配统计日志
   - 数据库写入失败不阻止分析结果返回
   - 过滤空项目

4. **增强** `src/routes/analysis/[id]/+page.svelte`
   - 添加分析摘要面板（匹配/未匹配/价格偏高统计）
   - 添加空项目状态提示
   - 改进分类颜色映射

### 验证方法
1. 重新构建后访问应用
2. 在控制台查看 `[VetLens]` 日志 — 应看到 FTS5 加载成功的消息
3. 上传账单或手动输入"血常规 ¥120" → 点击解读
4. 结果页应显示 ✅ 已匹配（置信度 > 60%），解释内容非空

## 问题四：OCR 提取项目混杂噪声行

> 发现时间: 2026-07-07 | 状态: ✅ 已修复

### 现象

OCR 识别账单后，提取结果混杂了大量噪声：

| 行 | OCR 原文 | 说明 |
|---|---|---|
| `No.: S001150722224614 1D: 53744` | 被提取为"收费项目 ¥53744" | 收据号被误判为诊疗项目 |
| `应付总额；因本到 935` | 被提取为"收费项目 ¥935" | 合计行混入 |
| `银行卡付:,00 本 935` | 被提取为"收费项目 ¥935" | 支付行混入 |
| `底部合计 56529.00` | 被提取为"收费项目 ¥56529" | 小票底部合计混入 |
| `急诊挂号` | 丢失 | 地址模式误匹配（`号`字） |
| `CR` | 丢失 | 拉丁字母缩写在中文账单中被归类为噪声 |

### 根因分析

原 `postProcessOcrText()` 只有简单的金额行尾正则，没有行分类逻辑：

- 无收据号/编号行的识别
- 无合计/支付行的识别
- 无地址/电话行的识别
- 无分类标题（如"检查费"/"药品费"）与具体项目的区分
- 纯拉丁字母项目（CR/X光/B超）被纯符号检测误杀
- 汉字间空格破坏模式匹配

### 修复内容

**重写** `src/lib/server/ocr/index.ts` — 新增四遍扫描管线：

1. **行分类器** `classifyLine()` — 将每行分为 7 种类型：

| 类型 | 特征 | 示例 |
|---|---|---|
| `header` | 医院/动物/宠物/诊所 | `北京爱宠动物医院` |
| `meta` | 地址(含号/路/区)、电话、支付信息 | `银行卡付`、`朝阳区XX路` |
| `section_title` | 分类标题，无具体项目名 | `检查费`、`药品费` |
| `total` | 合计/总计/应付/实收 | `应付总额 935` |
| `date` | 日期格式 | `2026-07-07` |
| `item` | 中文名+金额 / 英文缩写+金额 | `血常规 ¥80`、`CR 200` |
| `noise` | 纯符号、单字符、收据编号 | `No.: S001...` |

2. **行合并** — 处理项目名跨行换行（OCR 有时把金额输出到单独一行）
3. **结构化提取** — 从 item 行提取"项目名 + 金额"对
4. **后过滤** — 排除分类标题/地址/电话/合计行的漏网之鱼

5. **文本归一化** — 预处理阶段合并汉字间空格、修复小数点断列：
   ```typescript
   .replace(/([一-鿿]) ([一-鿿])/g, '$1$2')  // "一 次 性 耗 材" → "一次性耗材"
   .replace(/(\d)\s+\.\s*(\d)/g, '$1.$2')    // "150. 00" → "150.00"
   .replace(/(\d),\s+(\d)/g, '$1,$2')        // "100, 00" → "100,00"
   ```

6. **OCR 错字纠正字典** `correctOcrErrors()` — 修复常见识别错误，使用 `break` 防止级联替换：

| OCR 输出 | 纠正为 |
|---|---|
| 生化全讲 | 生化全套 |
| 血常规检 | 血常规 |
| 阿莫西材 | 阿莫西林 |
| 白细泡 | 白细胞 |

### 验证方法

测试用例（纯规则引擎，无需 LLM）：

```
输入: "一 次 性 耗 材 5\n血 常 规\n80\nCR 2 200"
       ↓ 归一化 + 分类 + 合并
输出: 一次性耗材 ¥5, 血常规 ¥80, CR ¥200 (x2)
```

API 测试：`POST /api/ocr/extract` → 纯规则引擎路径，返回干净列表。

## 问题五：LLM 智能提取臆造项目名

> 发现时间: 2026-07-07 | 状态: ✅ 已修复

### 现象

调用 DeepSeek API 提取 OCR 项目时，LLM 持续将项目名替换为不相关的医学词汇：

| OCR 原文 | 期望 | DeepSeek 返回 |
|---|---|---|
| 急诊挂号 ¥150 | 急诊挂号 | **留置针** ¥150 |
| 诊费 ¥20 | 诊费 | **雾化** ¥20 |
| 血检 ¥60 | 血检 | **血常规** ¥60 |

LLM 倾向于"理解并重写"而非"校对并保留"，通用聊天模型不适合做 OCR 后处理校对。

### 根因分析

1. DeepSeek-chat 作为通用对话模型，天然倾向于语义理解和改写，即使 prompt 反复强调"不要重命名"仍会臆造
2. 将原始 OCR 全文喂给 LLM 时，LLM 能从上下文中"猜"出上下文符合的医学名词，但这正是问题所在——它猜错了
3. 多次调整 prompt 均无效，LLM 的创造性与 OCR 校对的保守性之间存在根本矛盾

### 修复内容

**架构调整**：LLM 从主路径降级为极端兜底，规则引擎作为主方案。

```
原架构: OCR 文本 → LLM 提取（主路径）
新架构: OCR 文本 → 规则引擎（主路径）→ 提取成功 ✓ 返回
                  → 提取失败（0 项）→ LLM 兜底
```

`llmExtractItems()` 现在的行为：

1. **主路径**：调用 `postProcessOcrText()` 规则引擎 — 稳定、不臆造
2. **兜底路径**：仅当规则引擎提取到 0 个项目时才调用 LLM
3. **LLM 路径**也只发送纯文本的最后一个 fallback prompt，且不依赖 LLM 的名称纠正

### 验证方法

正常账单通过规则引擎 100% 提取，不再触发 LLM 路径。LLM 只在极端情况（如OCR输出全是乱码）时作为兜底。

## 问题六：OCR 输出字符间空格和跨行金额导致字段缺失

> 发现时间: 2026-07-07 | 状态: ✅ 已修复

### 现象

| 场景 | OCR 输出 | 问题 |
|---|---|---|
| 空格断字 | `一 次 性 耗 材 5` | 模式匹配失败，"一次性"和"耗材"被拆成两行 |
| 金额跨行 | `血常规\n80.00` | "血常规"无金额→暂存，"80.00"被NOISE_PATTERN吞噬→配对失败 |
| 逗号残留 | `诊费 1 20.00 20, 0.00` | cleanItemName 剥离数字后残留逗号→`诊费 1,`→数字"1"被逗号阻挡无法剥离 |

### 根因分析

1. **空格断字**：Tesseract OCR 对低分辨率/模糊的中文图片输出单字间空格，破坏`classifyLine`的完整中文词匹配
2. **金额跨行**：纯数字行`80.00`被 `NOISE_PATTERN` 匹配（`/^[\d\s\.,...]+$/`），分类为 `noise`，不会再被提取循环处理
3. **逗号残留**：`cleanItemName` 的数字剥离是单次 regex，剥离 `20.00` 后逗号残留，阻挡后续 `1` 的剥离

### 修复内容

1. **汉字间空格归一化** — `classifyLine` 和 `postProcessOcrText` 入口都添加：
   ```typescript
   .replace(/([一-鿿]) ([一-鿿])/g, '$1$2')  // 单空格
   .replace(/([一-鿿])\s+([一-鿿])/g, '$1$2') // 多空格
   ```

2. **纯金额行识别** — `classifyLine` 中纯金额行在噪声检测之前被拦截：
   ```typescript
   if (/^\s*\d{1,6}(?:[.,]\d{1,2})?\s*$/.test(trimmed)) return 'item';
   ```

3. **跨行金额配对** — 提取循环新增 `isAmountOnly` 检测：
   - 纯金额行 + 有暂存的项目名 → 自动配对并推入 items
   - 纯金额行 + 无暂存的项目名 → 跳过（孤立数字）

4. **逗号先行清理** — `cleanItemName` 在数字剥离前先清除尾部标点：
   ```typescript
   name = name.replace(/[,;；，、]+$/, '');
   ```
   数字剥离改为 `while` 循环，直到尾随数字全部清除。

### 验证方法

```
测试1: "一 次 性 耗 材 5"           → 一次性耗材 ¥5         ✅
测试2: "血常规\n80.00"              → 血常规 ¥80            ✅
测试3: "一 次 性 耗 材 5\n血 常 规\n80\nCR 2 200" → 3项全对 ✅
```
